# Multi-Log Analyzer (MVP) - Software Requirements Specification

## 1. Purpose

Разработать кросс-платформенное приложение для анализа нескольких лог-файлов одновременно.

Основная цель приложения — упростить анализ распределенных процессов, записывающих данные в разные лог-файлы, путем синхронной навигации между ними по времени и отображения нескольких логов одновременно.

---

# 2. Scope

Первая версия (MVP) должна обеспечивать:

* открытие отдельных файлов;
* открытие директорий с логами;
* одновременное отображение нескольких логов;
* синхронизацию логов по времени;
* поиск по файлам;
* автоматическое отображение новых строк в изменяемых логах;
* сохранение и восстановление пользовательской сессии.

В MVP не входят:

* SSH-доступ;
* файловый менеджер;
* фильтрация логов;
* подсветка логов;
* сохранение наборов фильтров;
* рекурсивный поиск по директориям.

Архитектура должна предусматривать добавление этих возможностей в будущих версиях.

---

# 3. Glossary

| Term             | Definition                                                                 |
| ---------------- | -------------------------------------------------------------------------- |
| Log Pane         | Независимая панель просмотра логов                                         |
| Active Pane      | Log Pane, имеющая текущий пользовательский фокус                           |
| Time Anchor Pane | Log Pane, относительно которой выполняется синхронизация остальных панелей |
| File Source      | Log Pane, открытая на отдельный файл                                       |
| Directory Source | Log Pane, открытая на директорию                                           |
| Timestamp        | Распознанная временная метка строки                                        |
| Viewport         | Область строк, отображаемая пользователю                                   |
| Session          | Сохраненное состояние открытых Log Pane                                    |

---

# 4. Functional Requirements

# 4.1 Opening Files

### FILE-001

Пользователь должен иметь возможность открыть отдельный файл.

### FILE-002

Имя файла должно отображаться в заголовке Log Pane.

### FILE-003

Содержимое файла должно отображаться начиная с первой строки.

### FILE-004

Если файл изменяется после открытия, приложение должно автоматически отображать новые строки.

### FILE-005

Приложение работает в режиме полного просмотра файла и отображения новых строк.

Поведение не должно ограничиваться последними строками файла.

---

# 4.2 Opening Directories

### DIR-001

Пользователь должен иметь возможность открыть директорию.

### DIR-002

Учитываются только файлы верхнего уровня директории.

### DIR-003

Поддиректории должны игнорироваться.

### DIR-004

В заголовке Log Pane должно отображаться имя директории.

### DIR-005

Под именем директории должно отображаться имя текущего выбранного файла.

### DIR-006

После открытия директории автоматически выбирается самый новый файл согласно активной стратегии сортировки.

---

# 4.3 Directory Navigation

### NAV-001

Directory Source должна содержать кнопки Previous Log и Next Log.

### NAV-002

Кнопки должны переключать отображаемый файл внутри директории.

### NAV-003

Список файлов должен формироваться только из файлов верхнего уровня директории.

### NAV-004

По умолчанию файлы должны сортироваться по Creation Time (Birth Time).

### NAV-005

Если Creation Time недоступен, должна использоваться резервная стратегия сортировки по имени файла.

### NAV-006

Если пользователь находится на последнем файле списка, кнопка Next Log должна быть неактивна.

### NAV-007

При появлении нового файла список должен быть пересчитан согласно текущему порядку сортировки.

### NAV-008

Текущий файл не должен автоматически переключаться при появлении новых файлов.

### NAV-009

При появлении нового файла должна обновляться доступность кнопок навигации.

### NAV-010

Если файл был удален и затем создан заново с тем же именем, он должен считаться новым файлом.

---

# 4.4 Log Pane Layout

### UI-001

Пользователь должен иметь возможность создавать новые Log Pane.

### UI-002

Кнопка добавления новой Log Pane должна отображаться справа от крайней правой панели.

### UI-003

При добавлении новой Log Pane должна разделяться крайняя правая панель.

### UI-004

После разделения новая панель должна занимать половину ширины правой панели.

### UI-005

Пользователь должен иметь возможность изменять размеры Log Pane перетаскиванием границ.

### UI-006

Измененные пользователем размеры должны сохраняться до следующего изменения.

### UI-007

При закрытии панели освободившееся пространство должно перераспределяться между оставшимися панелями.

### UI-008

Каждая Log Pane должна иметь собственную горизонтальную прокрутку.

### UI-009

Количество Log Pane не ограничивается фиксированным значением.

### UI-010

При недостатке горизонтального пространства контейнер должен поддерживать горизонтальную прокрутку.

---

# 4.5 Empty State

### UI-011

Если отсутствуют открытые Log Pane, в центре окна должна отображаться кнопка открытия файлов или директорий.

### UI-012

Должна поддерживаться загрузка файлов и директорий через Drag & Drop.

---

# 4.6 Search

### SEARCH-001

Каждая Log Pane должна иметь собственный независимый поиск.

### SEARCH-002

Поиск должен выполняться по полному содержимому файла независимо от текущей области отображения.

### SEARCH-003

Поиск должен поддерживать:

* обычный текст;
* регулярные выражения;
* учет регистра.

### SEARCH-004

Результаты поиска должны обновляться после появления новых строк.

---

# 4.7 Timestamp Recognition

### TIME-001

Форматы временных меток должны храниться во внешнем конфигурационном файле.

### TIME-002

Каждый формат должен содержать:

* шаблон обнаружения;
* шаблон разбора.

Пример:

```yaml
pattern: '\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}\.\d{3}'
parser: '%Y-%m-%d %H:%M:%S.%f'
```

### TIME-003

Поддерживается произвольное количество пользовательских форматов.

### TIME-004

Временная метка может находиться в любой части строки.

### TIME-005

Если строка содержит несколько временных меток, должна использоваться первая успешно распознанная временная метка.

### TIME-006

Если временная метка не может быть преобразована в корректную дату и время, она должна быть проигнорирована.

### TIME-007

Строка без корректной временной метки считается строкой без времени.

---

# 4.8 Time Synchronization

### SYNC-001

Должна существовать настройка Synchronize Scrolling.

По умолчанию настройка включена.

### SYNC-002

Time Anchor Pane определяется автоматически как Active Pane.

### SYNC-003

Time Anchor Pane изменяется при взаимодействии пользователя со скроллом, поиском или переходом по файлу.

### SYNC-004

Для каждой строки определяется значение:

timestamp(line)

### SYNC-005

При изменении позиции Time Anchor Pane до времени T остальные синхронизируемые панели должны переходить к строке:

max(timestamp <= T)

### SYNC-006

Log Pane без распознанных временных меток должна автоматически исключаться из механизма синхронизации.

### SYNC-007

Log Pane без временных меток не должна влиять на другие панели.

### SYNC-008

Другие панели не должны влиять на Log Pane без временных меток.

### SYNC-009

При отключении Synchronize Scrolling каждая Log Pane должна прокручиваться независимо.

---

# 4.9 Time Offset

### OFFSET-001

Для каждой Log Pane должно поддерживаться собственное временное смещение.

### OFFSET-002

Смещение должно задаваться в:

* днях;
* часах;
* минутах;
* секундах;
* миллисекундах.

### OFFSET-003

Смещение должно учитываться при синхронизации по времени.

---

# 4.10 File Deletion

### FILE-DELETE-001

После удаления открытого файла Log Pane не должна закрываться автоматически.

### FILE-DELETE-002

Уже загруженное содержимое должно оставаться доступным.

### FILE-DELETE-003

Поиск по уже загруженному содержимому должен продолжать работать.

### FILE-DELETE-004

Получение новых данных должно быть прекращено.

### FILE-DELETE-005

Интерфейс должен отображать статус удаленного файла.

---

# 4.11 Log Rotation

### FILE-ROTATE-001

Если файл был заменен новым файлом с тем же именем, приложение должно считать его новым файлом.

### FILE-ROTATE-002

Открытая Log Pane должна автоматически переключаться на новый файл.

---

# 4.12 Text Selection

### TEXT-001

Пользователь должен иметь возможность выделять текст.

### TEXT-002

Пользователь должен иметь возможность копировать выделенный текст:

* через горячие клавиши;
* через контекстное меню.

---

# 4.13 Session Restore

### STATE-001

После перезапуска приложения должны восстанавливаться:

* открытые Log Pane;
* порядок Log Pane;
* размеры Log Pane;
* открытые файлы;
* открытые директории;
* выбранный файл внутри Directory Source;
* активный набор фильтров и подсветок (если функциональность появится в будущих версиях).

### STATE-002

Позиция прокрутки не должна восстанавливаться.

---

# 5. Future Features (Not MVP)

## FILTER-002

Негативная фильтрация по тексту и регулярным выражениям.
Нужно скрывать все строки которые содержат какую-либо подстроку из негативных фильтров. 

## FILTER-003

Фильтрация одновременно во всех открытых Log Pane.

## FILTER-004

Боковая панель управления фильтрами.

## HL-001

Подсветка совпадений.

## HL-002

Поддержка нескольких цветовых схем.

## HL-003

Приложение содержит 5 готовых шаблонов.

## HL-004

Настраиваемые foreground/background цвета.

## HL-005

Поддержка стилей текста.

## HL-006

Сохранение и загрузка наборов подсветок.

## HL-007 

Управление стилями подсветок расположено совместно с управлением фильтрами, на одной панель друг под другом.

## SEARCH-DIR-001

Поиск по всем файлам открытых директории во всех Log Pane.

## SEARCH-DIR-002

Навигация к найденным результатам.

## SEARCH-DIR-003

Меню для поиска по файлам расположено в отдельной боковой панели.

## SSH-001

Открытие файлов через OpenSSH.

## SSH-002

Поддержка парольной аутентификации.

## SSH-003

Поддержка системного OpenSSH Config.

# 6. Performance Requirements

### PERF-001

Открытие файла размером 20 MB должно занимать не более 1 секунды.

### PERF-002

Поиск по файлу размером 20 MB должен выполняться не более 1 секунды.

### PERF-003

Переключение между файлами должно занимать не более 200 мс.

### PERF-004

Приложение должно использовать виртуализацию отображения строк.

### PERF-005

Подсветка и фильтрация должны применяться только к области:

N строк выше viewport
+
viewport
+
N строк ниже viewport

### PERF-006

Размер буфера предварительной обработки должен настраиваться.

### PERF-007

При недостатке памяти операция открытия файла должна быть остановлена до загрузки содержимого.

---

# 7. File Size Limits

### LIMIT-001

Перед открытием файла приложение должно проверять размер файла.

### LIMIT-002

Максимальный размер файла должен настраиваться пользователем.

### LIMIT-003

Значение по умолчанию — 20 MB.

### LIMIT-004

Если файл превышает допустимый размер, пользователю должно отображаться сообщение об ошибке.

---

# 8. Character Encodings

### ENC-001

Приложение должно автоматически определять кодировку файла.

### ENC-002

Обязательные поддерживаемые кодировки:

* UTF-8
* UTF-8 BOM
* UTF-16 LE
* UTF-16 BE
* Windows-1251
* Windows-1252

### ENC-003

Если кодировка не определена автоматически, пользователь должен иметь возможность выбрать ее вручную.

---

# 9. Reliability Requirements

### REL-001

Поврежденный лог-файл не должен приводить к аварийному завершению приложения.

### REL-002

Ошибка чтения одного файла не должна влиять на работу остальных Log Pane.

### REL-003

Необработанные исключения не должны приводить к потере пользовательской сессии.

---

# 10. Security Requirements

### SEC-001

Приложение не должно изменять содержимое открываемых файлов.

### SEC-002

Приложение не должно выполнять команды, обнаруженные в логах.

### SEC-003

Приложение должно работать в режиме только чтения относительно анализируемых файлов.

---

# 11. Architecture Requirements

### ARCH-001

Приложение должно использовать единую кодовую базу для Web и Desktop версий.

### ARCH-002

Бизнес-логика должна реализовываться один раз и использоваться всеми платформами.

### ARCH-003

Платформозависимый код должен быть изолирован через интерфейсы.

### ARCH-004

Повторное использование UI-компонентов между платформами должно быть максимальным.

### ARCH-005

Навигация по файлам должна использовать навигационный индекс, формируемый на основании текущей сортировки.

### ARCH-006

Навигационный индекс должен обеспечивать получение:

* текущего файла;
* предыдущего файла;
* следующего файла.

### ARCH-007

При изменении состава директории навигационный индекс должен пересчитываться.

---

# 12. Platform Requirements

## Desktop

Технологический стек:

* Tauri 2
* Web Frontend
* Windows
* macOS
* Linux

Поддерживаются:

* локальные файлы;
* локальные директории;
* мониторинг изменений файлов;
* автоматическое отображение новых строк.

## Web

Поддерживаются:

* Drag & Drop файлов;
* Drag & Drop директорий;
* File System Access API (если доступен);
* загрузка файлов через браузер.

Не поддерживаются:

* мониторинг локальной файловой системы;
* автоматическое обнаружение новых файлов;
* SSH-доступ.
